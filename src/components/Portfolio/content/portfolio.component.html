My MEAN-Stack project

Below is the summary of this mean stack app I Build. I will explain the technologies and why I used them

MongoDB

Why MongoDB?

The reason MongoDB was such a good candidate for this project is because it has a lot of data. Separating this data and normalizing it into different tables would not only make things more complicated but also slow down the requests to the database. Instead I thought it would be better to have all the data in BSON documents which I can easily access and convert to JSON using a library.

Mongoose

To connect the Express Api to the database I used the Mongoose Library which uses models to create schemes for a doucment and has an ORM approach See: Server/models

the data

One of the biggest problems in this project was to get access to a lot of data to simulate the stats coming fro a real game. to solve this problem I wrote a script that copied the data from an unofficial Overwatch API and restructured it so it would fit inside the MongoDB document schemes that I created. See: Server/dbfill.js

Express Api

Why Node.js?

The reason Node.js is such a good candidate for an API is that it is very easy to write asynchronous calls + the fact that Javascript already uses JSON for objects. These 2 things make Node.js the ideal candidate. Node.js is also very scalable and the requests remain idle when it is not used or waiting for I/O.

Why Express Framework

Express is an framework build on Node.js is because Express makes it very easy to create an REST-api. It is also the biggest and longest maintained Node.js framework.

The API

The server

The beautiful thing about Node.js is that you can write a server in just 4 lines of code. The server this application is using is a bit larger than 4lines of code but still a very compact server which can be easily run with the Node.js-CLI(NPM) It also ues several middlewares to make things easier like bodyparser whic is responeible for easy access to the request-body on HTTP-requests. See Server/server.js

Security

An important part of an api is security cause when you allow third-party applications to access the Api you need to enable CORS(Cross Origin Requests). So for extra security I added Authentication to the API. Which makes sure that the person/application who accesses the API is known to the provider of the API this evens enables you to log everything that happens on the API. The authentication is done with JSON Web tokens. Everytime the user logs in he gets a token which can be used for a 24h period to access the API. every time the user makes an API call the token must be one of the request headers or the requests get blocked by the middleware and a "no token provide" message is sent to the client. All Accounts are also stored in the MongoDB database, passwords are encrypted with the BCrypt algorithm. See Server/functions/encrypt.js

API requests

All API requests are configured in the apirouter.js file. They are all programmed so that they word as asynchronous as possible. Most of the routes are secured by the JWT token middleware. All the request return data in JSON in a global format which has a msg,a succes value an ofcourse a data value. Also the data is remodeled into an JSON object because it would be insecure to send all data from our database to the client. See Server/router/apirouter.js

Webpack

Webpack is an module bundler which makes it easy to bundle all the different scripts (which are a lot with angular 2) to a single script which is included in the application.

Typescript

Angular 2 is the easiest when written in Typescript. Typescript is a superset of javascript that adds types to Javascript when compiled you get plain Javascript.

The advantage of components is that every component has it task an does that task only creating a very lose-coupled architecture. these components can interact which each other as seen in the Compare page where the searchbar outputs events to the compare component or in the Api components which passes FormGroups into each api-example. Angular2 Services

Services are like the logic layer in the Angular framework they can be usefull for a lot of things in our application they are mostly used to make asynchronous requests to the API. When the data is received it is send to the UI components which will receive it if they are subscirbed to that service.

Angular2 router

The router in Angular 2 is different than any other router this is because the whole application is just one page. This means that no GET request is send to the server, all the routing is done with Javascript this also explains why you get a 404 error when you try to refresh a page.
